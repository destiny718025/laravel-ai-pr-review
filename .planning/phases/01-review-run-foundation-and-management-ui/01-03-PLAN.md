## Phase Goal

**As a** local/private Laravel AI PR Review user, **I want to** submit a GitHub pull request URL and see a persisted review run in the management UI, **so that** I can prove the review workflow foundation works before adding GitHub fetching and AI analysis.

<objective>
Build the no-login `/reviews` route, controller, create dashboard, and minimal detail shell on top of the service foundation.

Purpose: Let the user submit a GitHub pull request URL through a management UI and receive either a created pending review run or a structured service validation message.
Output: Controller-backed routes, dashboard layout/view, minimal detail shell, status component, and feature tests for redirect, route-follow, form submission, valid creation, and invalid no-record behavior.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser form to ReviewController | Untrusted pr_url input enters the HTTP layer. |
| ReviewController to ReviewRunService | HTTP input is handed to the business workflow boundary. |
| Session flash to Blade view | Service messages and stable codes are rendered without raw exceptions. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-01-03-01 | Tampering | POST /reviews | medium | mitigate | Controller requires pr_url and delegates semantic validation to ReviewRunService before persistence. |
| T-01-03-02 | Information Disclosure | resources/views/reviews/index.blade.php | medium | mitigate | Render only service user-facing messages and diagnostic error codes; never render stack traces or raw provider data. |
| T-01-03-03 | Spoofing | No-login management UI | low | accept | Phase 1 is explicitly local/private and has no auth requirement; later auth is outside this phase. |
</threat_model>
