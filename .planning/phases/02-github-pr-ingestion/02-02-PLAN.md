## Phase Goal

**As a** reviewer using a saved review run, **I want to** fetch GitHub pull request metadata and changed files from the detail page, **so that** the application stores a reusable snapshot for later AI review work.

<objective>
Deliver the first end-to-end GitHub ingestion slice: manual fetch on the existing detail page, snapshot persistence, and visible fetched state, per D-04, D-05, D-06, D-07, D-08, and D-09.

Purpose: turn the client seam from Plan 02-01 into a real user action that stores GitHub data for future phases without expanding into hunk parsing or queued AI work.
Output: a fetch route/controller action, ingestion service/result object, review-run snapshot storage, `review_run_files` persistence, and happy-path feature coverage.
</objective>

Scope note: this plan intentionally stays as one vertical slice despite touching UI, controller, service, repository, model, migration, and test files, because the user-visible `Fetch` workflow is only real when the route, persistence, and detail-page rendering land together. Scope risk is constrained by Plan 02-01 already establishing the GitHub client seam, Task 1 forcing the happy-path contract first, and Task 2 keeping all database writes inside `ReviewRunRepository` per D-04 through D-09 instead of spreading persistence logic across layers.
