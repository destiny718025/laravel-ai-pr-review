## Phase Goal

**As a** reviewer running queued AI analysis, **I want to** reject malformed AI output safely, **so that** only well-formed findings can move toward persistence and the run shows stable failure messages when something goes wrong.

<objective>
Add the structured finding DTOs, validator, failure mapper, and execution-service integration that enforce safe JSON validation and safe failure behavior per D-10, D-14, and D-15.

Purpose: separate the provider seam from the validation boundary so malformed AI output is blocked before persistence or UI work begins.
Output: `ValidatedFindingPayload`, `AIReviewFailure`, `AIReviewPayloadValidator`, `AIReviewFailureMapper`, execution-service validation wiring, and passing unit/feature coverage.
</objective>

|-----------|----------|-----------|----------|-------------|-----------------|

| T-03-08 | Information Disclosure | safe error summaries | high | mitigate | `AIReviewFailureMapper` must whitelist human-safe messages and exclude raw provider payloads, headers, and secrets. |
| T-03-09 | Repudiation | execution failure transitions | medium | mitigate | Route validation and provider failures through repository-owned failed-state transitions so timestamps and safe summaries are deterministic. |
| T-03-SC | Tampering | npm/pip/cargo installs | low | accept | No package-manager installs are introduced; validation and failure mapping use existing Laravel capabilities only. |
